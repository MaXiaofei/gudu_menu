## 1. 项目概述

「烟火小食单」是一个面向**家庭自用**的菜谱与膳食管理系统，含一个 Web 管理后台（「烟火管理后台」）和一个微信小程序（「小食单」）。

**核心价值**：在家庭日常菜谱 / 食材 / 菜单 / 备菜采购管理的基础上，**为家中的特殊人群（三高、宝宝辅食等）提供营养定制菜单能力**——普通人正常排菜吃，只有特殊人群才走营养约束那套；并逐步引入 AI 与健康趋势闭环，辅助特殊人群的饮食管理。

---

## 2. 目标用户与核心场景

- **使用者**：掌勺 / 备菜人（操作者，本项目作者本人）+ 家庭成员（吃饭的人，含需照顾的特殊人群）。
- **核心场景**：
  1. 日常管理菜谱 / 食材 / 菜单，按周排菜。
  2. 备菜与采购：周计划 → 按时间范围（今日 / 3 天 / 本周 / 自定义）生成采购清单（去重、按品类分区）→ 库存管理。
  3. 健康定制：为三高 / 宝宝等特殊人群，按营养约束（低糖、低 GI、低脂、控钠等）定制菜单。
  4. 成本控制：按用餐预算排菜单。
  5. 健康闭环：记录每日实际摄入 + 健康指标趋势，反馈调整推荐。

## 3. 设计铁律

> 用户明确要求的原则，全系统遵守。

- **能不写死就配置化**：凡可枚举的维度，全部做成配置中心的字典，可后台增删改，不写死代码（含营养指标、点评维度、健康指标等「看似固定」的项）。

> 注：核心实体该有的 CRUD 仍会做，但按需决定。

## 5. 功能分期

> 原则：先把数据地基和流程跑通、能用起来，再叠加智能。AI 全部 V2。

### 🟢 MVP — Web 后台，数据地基 + 基础流程（不接 AI）

- **账号**：主账号登录（操作者），家庭成员作为档案管理。
- **家庭成员 + 健康档案**：每人挂静态档案（身高体重、过敏 / 忌口、特殊人群标签、营养约束上限）。
- **配置中心**：第 4 节全部字典的增删改 + 种子数据。
- **食材库**：营养数据（**可配置指标维度，EAV 结构**）、价格、采购品类、**基础库存**；记录购买 / 食用次数。
- **菜品库**：菜名 / 做法 / **步骤（文字 + 图片，可多张）** / 备注 / 封面 / **备料时间 + 烹饪时间** / 价格 / **难度 1–5**；多对多挂「菜系 / 标签 / 分类 / 食材（带用量重量）」；**每道菜展示营养信息**（由食材聚合）；**按份数缩放用量**；**改动保留历史版本，可删单个 / 多个历史**。
- **菜单管理**：菜单种类；**手动排菜单**；菜品价 → **菜单总价**；按份数 / 人数算总量与营养。
- **菜库基础使用**：收藏、做过标记、搜索 + 多维筛选（菜系 / 标签 / 营养 / 时间 / 难度）。
- **基础烹饪记录**：记录做过哪些菜。
- **数据备份 / 恢复**：全量数据导出 / 导入（防丢）。

### 🟡 V1 — 非智能业务增强 + 小程序端

- **拖拽式周历排菜**：菜拖到具体某天 / 某餐（早中晚加餐），随时调整。
- **菜单 / 周计划模板复用**：常用一周菜单存模板，一键套用。
- **排菜去重提醒**：同周期重复某菜时提醒。
- **采购清单**：从周计划**按时间范围**（今日 / 3 天 / 本周 / 自定义）**自动生成** + **去重合并**（多菜同食材合并重量）+ **按采购品类分区** + 预估价 + 勾选已买。
- **食材库存管理**：余量、**临期 / 不足提醒**（不做条码扫描）。
- **按食材反向找菜**：勾选冰箱现有食材，推荐能做的菜（顺带清库存）。
- **菜单时间并行排程**：基于各菜备料 / 烹饪时间 + 灶台并行度 N，算菜单总时长；**提供两种模式 tab**：① 备料·烹饪交错并行；② 先备料后烹饪、不交错。
- **自定义菜单**：按口味 / 场景 / 人群 / 预算手动多维组合筛选。
- **小程序端**：点菜、**点评（星级 + 文字 + 图片 + 多维度打分）**、预制菜单、烹饪记录、备菜 / 采购清单、**烹饪步骤计时器**、**录入新菜品**（家庭成员可贡献菜库）。
- **菜谱导入**：**网页链接解析**（粘贴下厨房 / 美食杰等链接，自动抓成结构化菜谱）+ 批量 JSON 导入。
- **菜谱导出**：PDF / 图片（分享打印）。
- **采购清单导出**：CSV / 纯文本（去超市、发微信）。
- **通知触达**：站内提醒（消息中心，主力）+ 小程序订阅消息（关键提醒）；「通知中心」统一管理，每类提醒可配通道 + 开关。
- **权限边界**：后台对每个成员分配**小程序功能权限**——角色标签做默认权限模板（掌勺=全权，普通成员=只能点评+看菜单）+ 个人功能勾选微调（增 / 改 / 删菜品、排菜、采购、看家人健康数据）。轻量功能开关矩阵，不做重型 RBAC。
- **每日饮食记录（手工）**：记录每天实际吃了什么、摄入量（为 V2 AI 分析攒数据；本阶段不带分析）。
- **账号体系**：主账号登录，小程序里可切换「当前就餐成员」。

### 🔴 V2 — AI 智能 + 健康闭环

- **AI 营养数据自动补全**：输入食材 / 菜名，AI 填充营养数据。
- **AI 智能菜单推荐**：综合健康档案 + 口味偏好（购买 / 食用历史）+ 预算 + 营养约束，推荐每日 / 每周菜单。
- **AI 营养顾问对话**：聊天式提问（如「今天血糖偏高晚饭吃啥」），结合家人档案给建议。
- **AI 菜品营养估算（含拍照识别）**：拍一道菜 / 一餐，AI 估算营养构成。
- **AI 食材替换建议**：缺料时推荐替代（如黄油 ↔ 植物油）。
- **健康指标趋势追踪**：血糖 / 血压 / 血脂 / 体重的长期记录 + **趋势曲线图**（健康档案从「静态值」升级为「时间序列」）。
- **每日营养目标 + 达成追踪**：每人每天热量 / 糖 / GI / 钠等目标，饮食日记对照、超限提醒。
- **饮食日记 AI 分析**：基于 V1 已记录的每日摄入，AI 结合营养目标做分析、超限提醒，形成「推荐 → 吃 → 记录 → 看趋势 → 调整」闭环。

## 6. 技术架构与技术栈

> 用户已定核心栈，其余为补充建议。

| 层 | 组件 | 选型 | 状态 |
|---|---|---|---|

| 持久层 | MyBatis **+ MyBatis-Plus** | CRUD 增强 | ✅ MyBatis 定 / 💡 Plus 建议 |
| 数据库 | MySQL 8 | 主库 | ✅ 用户定 |
| 缓存 | Redis | 营养聚合 / 会话 / 限流 | 💡 补充建议 |
| 认证 | **Sa-Token（有状态 token + Redis）** | 不用 JWT 无状态 | ✅ 已定 |
| 接口文档 | Knife4j | Swagger 增强 | 💡 补充建议 |
| Web 后台 | Vue3 + TypeScript + Vite | 后台前端 | ✅ Vue3 定 / 💡 其余建议 |
| 后台 UI | Element Plus + Pinia + ECharts | 管理后台 + 图表（健康趋势 / 营养图） | 💡 补充建议 |
| 小程序 | uniapp（Vue3）+ uView | 多端 | ✅ uniapp 定 / 💡 补 UI 库 |
| AI / 数据服务 | **Python + FastAPI** | 网页菜谱抓取解析 + V2 的 LLM / 营养 / 拍照识别；与 Java 主服务 HTTP 通信 | ✅ 用户定 |
| AI 大模型 | **智谱 GLM** | 由 Python 服务调用 | ✅ 用户定 |
| 图片存储 | 本地起步 → **MinIO（自建对象存储，Docker 起）** | 菜谱 / 步骤 / 点评图；上云可切云 OSS | ✅ 已定 |
| 部署 | **Docker Compose** | 先本地自用，后上云（一台云服务器） | ✅ 先本地后云定 |
| 登录 | 后台账号密码 + 小程序微信登录 | Sa-Token 统一管两入口 token | ✅ 已定 |

**架构形态：前后端分离。**
- **前端**：Web 后台（Vue3）+ 小程序（uniapp）均为**独立前端工程**，通过 HTTP / Axios 消费同一套 Java RESTful API，各自独立构建部署。
- **后端**：单体（不拆微服务）——Java（Spring Boot）主业务服务 + Python（FastAPI）AI/数据服务 + MySQL + Redis，Docker Compose 一键编排；AI 相关请求由 Java 转发到 Python 服务。

**工程目录（三端分离，各自独立工程）**：`backend/`（后端接口：Java 主服务 + Python `ai-service`）/ `admin/`（Web 管理后台 Vue3）/ `miniapp/`（小程序 uniapp，V1 建）；三端共享同一套 Java RESTful API。

## 7. 系统模块设计

按「单一职责、独立可测」拆分：

1. **账号 / 认证模块**（Sa-Token）：主账号登录、微信登录、统一 token、会话管理。
2. **家庭成员模块**：成员档案、健康档案、角色标签、小程序功能权限。
3. **配置中心模块**：所有字典维度的增删改（第 4 节）。
4. **食材库模块**：食材、营养（EAV）、价格、采购品类、库存。
5. **菜品库模块**：菜品、多对多关联、**营养聚合计算**（纯函数）、份数缩放、步骤图文、**历史版本**。
6. **菜单模块**：菜单、菜单-菜品、成本汇总、时间排程（两种模式）。
7. **周计划排菜模块（V1）**：日历视图、拖拽排菜、去重提醒、模板复用。
8. **采购清单模块（V1）**：按时间范围生成、去重合并、品类分区、勾选、导出。
9. **库存模块（V1）**：余量、临期 / 不足提醒。
10. **点评模块（V1）**：星级 + 文字 + 图片 + 多维度打分。
11. **烹饪记录 / 步骤计时模块**：做过记录、烹饪倒计时。
12. **菜库使用模块**：收藏、做过、搜索筛选、反向找菜。
13. **导入导出模块**：网页链接解析导入、JSON 备份恢复、PDF/图片/CSV 导出。
14. **通知中心模块（V1）**：站内提醒 + 小程序订阅消息，多通道可配。
15. **AI 模块（V2，Python 服务）**：营养补全、菜单推荐、顾问对话、营养估算（含拍照）、食材替换。
16. **健康闭环模块（V2）**：健康指标趋势、营养目标、饮食日记。

> 模块间通过纯函数接口隔离依赖（营养聚合、规则约束、排程算法等），便于单测。

## 8. 数据模型（核心实体与关系）

> MyBatis schema 风格，字段为要点。带「维表」的均可配置。

### 账号 / 成员
- **user**：主账号（`username`, `password_hash`）。
- **member**：家庭成员（`name`, `role_tags[]`(chef/prep/member，可叠加), `health_profile`(身高、体重、过敏忌口、特殊人群标签), `mp_permissions[]`(小程序功能权限))。
- **health_record**（V2 趋势）：`member_id`, `metric_id`(健康指标维表), `value`, `recorded_at` —— 多次记录，画趋势。
- **nutrition_goal**（V2）：`member_id`, `metric_id`, `daily_limit`(每日上限 / 区间，如糖<25g、钠<2000mg、热量区间)。

### 配置中心（维表，均增删改）
- `cuisine`, `dish_tag`, `dish_category`, `menu_type`, `audience_tag`(特殊人群), `unit`(计量单位，标记称重/计数), `role_tag`, `purchase_category`(采购品类)。
- **nutrition_metric**（营养指标维表）：`name`, `unit`, `group`(宏量/微量)。
- **review_dimension**（点评维度维表）：`name`(口味/难度/营养均衡/外观…)。
- **health_metric**（健康指标维表）：`name`, `unit`(血糖/血压/血脂/体重…)。

### 食材
- **ingredient**：`name`, `unit_id`, `price`, `purchase_category_id`, `purchase_count`, `usage_count`。
- **ingredient_nutrition**（食材-营养指标-值，EAV）：`ingredient_id`, `metric_id`, `value`(per 100g)。
- **pantry**（V1 库存）：`ingredient_id`, `amount`, `unit_id`, `expire_date`, `low_threshold`(低库存阈值)。

### 菜品
- **dish**：`name`, `note`, `cover_url`, `prep_time`, `cook_time`, `price`, `difficulty`(1–5)。
- **dish_step**：`dish_id`, `seq`(序号), `text`, `images[]` —— 文字 + 图片分步。
- **dish_ingredient**（菜品-食材-用量重量）：`dish_id`, `ingredient_id`, `amount`。
- **dish_history**：菜品历史版本快照，支持删除单个 / 多个。
- 关联（多对多）：`cuisines[]`, `tags[]`, `categories[]`。
- *派生（纯函数）*：**菜品营养** = Σ(ingredient_nutrition.value × amount / 100)，按 metric 分组；**按份数缩放** = × serving_factor。

### 菜单 / 周计划
- **menu**：`name`, `type_id`, `target_member_id`, `serving_count`(份数/人数)。
- **menu_dish**：`menu_id`, `dish_id`, `serving_factor`。
- *派生*：菜单营养 / 总价 = Σ(菜品 × 份数)；**排程总时长** = 两种模式（交错并行 / 先备后做）按灶台并行度 **N=2（双灶）** 调度。
- **menu_template**（V1）：模板化菜单集合，可套用到周计划。
- **meal_plan**（V1 周历）：`week_start`。
- **meal_plan_item**（V1）：`plan_id`, `date`, `meal`(早/中/晚/加餐), `dish_id`, `serving_factor`。

### 采购清单
- **shopping_list**（V1）：`source_plan_id/menu_id`, `time_range`(今日/3天/本周/自定义), `created_at`。
- **shopping_item**（V1）：`ingredient_id`, `total_amount`(合并后重量), `purchase_category_id`(分区), `estimated_price`, `purchased`(bool)。

### 点评 / 记录 / 使用
- **review**（V1）：`member_id`, `dish_id/menu_id`, `star_rating`(1–5), `text`, `images[]`, `created_at`。
- **review_score**（V1 维度分）：`review_id`, `dimension_id`, `score`。
- **cooking_record**：`dish_id`, `member_id`, `cooked_at`, `note`（兼作「做过」历史）。
- **favorite**：`member_id`, `dish_id`。
- **daily_log**（V2 饮食日记）：`member_id`, `date`, `items[]`, `total_calorie`, `total_sugar`…（对照 nutrition_goal）。

### 通知
- **notification**（V1）：`member_id`, `type`(临期/采购/备菜), `channel`(站内/订阅消息), `title`, `content`, `read`, `created_at`。

## 9. 关键设计决策（已对齐）

| # | 决策 | 结论 |
|---|---|---|
| 1 | 项目定位 | 家庭自用；三高定制是**面向特殊人群**的专项能力，非全家通用 |
| 2 | 权限模型 | 角色标签（chef/prep/member，可叠加）+ 小程序功能勾选；不做 RBAC |
| 3 | 健康定制依据 | 每成员「健康档案 + 营养约束」 |
| 4 | 制作时间 | 精细（备料 + 烹饪）；菜单总时长并行排程，**两种模式 tab**（交错并行 / 先备后做），依赖灶台并行度 N |
| 5 | 营养数据 | **可配置指标维度（EAV）**，菜品级聚合 |
| 6 | 特殊人群 | 三高、宝宝辅食等，每道菜展示营养 |
| 7 | 定制机制 | 规则筛选 + AI 推荐，AI 全部 V2 |
| 8 | 成本 / 预算 | 菜品记价、菜单算总价、按预算筛选 / 推荐 |

## 12. 营养数据来源（已调研确定）

分层获取，免费优先：

| 数据 | 来源 | 用途 / 说明 |
|---|---|---|
| 常规营养（碳水/蛋白/脂肪/热量） | **《中国食物成分表第6版》JSON**（GitHub china-food-composition-data） | 起步种子数据，直接导入常用食材，中文权威 |
| 常规营养（扩充） | **USDA FoodData Central API**（免费，注册 Key，~38 万食物） | 批量拉取国际食材，需配中文名映射 |
| 常规营养（最全·可选付费） | **薄荷科学.AI**（ai.boohee.com，70 万食物 / 150+ 营养素） | 预算允许时接入，最全 |
| **GI 升糖指数** | **公共卫生科学数据中心**（200+ 中国常见食物）+ **chinafoodgi.com**（悉尼大学中国区授权） | GI 权威中文数据，手工录入常见项 |
| GI（国际补充） | 悉尼大学 SUGiRS（glycemicindex.com） | 国际权威，英文 |
| 缺失补全 | **智谱 GLM（V2）** | 推断补全，标注「AI 推断 / 可信度」，用户可校正 |
| 轻量备选 | TianAPI / 聚合数据 营养成分 API | 快速小规模接入 |

**落地策略**：
1. **MVP**：导入 GitHub《中国食物成分表第6版》JSON 作种子（碳水/蛋白/脂肪/热量）+ 公共卫生数据中心 GI（200+ 种）。
2. **V1**：接 USDA FoodData Central API 批量扩充食材库（配中文名映射）。
3. **V2**：智谱 GLM 补全缺失项 + 标可信度；必要时接薄荷 API。
4. **全程**：所有营养数据可手工修正；AI 补全数据单独标注来源 / 可信度。

**参考来源**：[USDA FoodData Central](https://fdc.nal.usda.gov/) ｜ [中国食物成分表 JSON](https://github.com/Sanotsu/china-food-composition-data) ｜ [公共卫生科学数据中心 GI](https://www.phsciinedata.cn/Share/ky_sjml.jsp?id=2ec166ba-85a5-460a-87ff-792827b7370a) ｜ [chinafoodgi.com](http://www.chinafoodgi.com/) ｜ [薄荷科学.AI](https://ai.boohee.com/)
